# Reports

Every GrafanaProbe test run generates two report artifacts: a **JSON report** (machine-readable) and an **HTML report** (human-readable). Both are stored in `backend/reports/` and persisted in the SQLite database.

---

## Report lifecycle

```
Test run starts
      │
      ▼
Test Engine executes categories in order
      │
      ▼ (streaming via WebSocket)
Results accumulate in memory
      │
      ▼
Run completes → JSON file written to backend/reports/
      │
      ▼
HTML report generated from JSON
      │
      ▼
Report indexed in SQLite (fast lookup by run ID)
```

---

## JSON report format

Reports are stored as `backend/reports/report-<timestamp>.json`.

### Top-level structure

```json
{
  "id": "report-1712345678901",
  "timestamp": "2024-04-05T14:22:58.901Z",
  "grafanaUrl": "http://grafana.internal:3000",
  "duration": 42318,
  "summary": {
    "total": 147,
    "passed": 132,
    "warned": 11,
    "failed": 4,
    "errors": 0,
    "categories": 17
  },
  "categories": [ ... ]
}
```

### Category structure

```json
{
  "id": "datasources",
  "name": "Data Sources",
  "icon": "🔌",
  "status": "warn",
  "duration": 1823,
  "tests": [
    {
      "name": "Health check: Prometheus",
      "status": "pass",
      "message": "Data source is healthy",
      "grafanaUrl": "http://grafana.internal:3000/connections/datasources/edit/abc123",
      "details": {
        "uid": "abc123",
        "type": "prometheus",
        "responseTime": 142
      }
    },
    {
      "name": "Health check: Loki",
      "status": "warn",
      "message": "Data source returned warning: no labels found",
      "grafanaUrl": "http://grafana.internal:3000/connections/datasources/edit/def456"
    }
  ]
}
```

### Status values

| Status | JSON value | Meaning |
|--------|-----------|---------|
| Pass | `"pass"` | Test passed |
| Warning | `"warn"` | Potential issue, non-blocking |
| Fail | `"fail"` | Issue requires attention |
| Error | `"error"` | Test could not execute |

## HTML report

The HTML report is a self-contained single-file document generated from the JSON report. It requires no external dependencies and can be opened in any browser or shared via email/Slack.

### HTML report sections

1. **Header** — Run timestamp, Grafana URL, duration, summary counts (PASS/WARN/FAIL/ERROR)
2. **Summary bar** — Visual breakdown of results by status
3. **Category results** — Expandable accordion for each of the 17 categories
4. **Individual test results** — Each test with status badge, message, and Grafana deep-link
5. **AI Analysis** (if available) — LLM-generated failure summary appended at the bottom

### Accessing HTML reports

**From the UI:**
1. Go to **Reports** in the sidebar
2. Find your run
3. Click the **HTML** button to open in a new tab

**Via API:**
```
GET /api/reports/html/report-1712345678901.html
```

**Direct file access:**
```bash
open backend/reports/report-1712345678901.html
```

## AI failure analysis

When an LLM provider is configured (OpenAI or Anthropic), you can request AI analysis of any failed run.

### What the AI receives

GrafanaProbe sends the LLM a structured prompt containing:
- Summary of FAIL and WARN results
- Category names and specific failure messages
- Grafana version and environment URL
- Count of affected resources

The raw Grafana API responses or dashboard content are **not** sent to the LLM — only the test result messages generated by GrafanaProbe.

### AI analysis output

The LLM returns:

```
## GrafanaProbe AI Analysis

### Failures detected
- 4 FAIL results across 2 categories (Data Sources, Alerts)

### Root cause analysis
**Data Sources (2 failures):**
The Loki data source health check is failing, likely due to the backend
Loki instance being unreachable. This is consistent with 0 log data in
dashboards that use Loki.

**Alerts (2 failures):**
Two alert rules are in Error state because they query the Loki data source.
These alerts will not evaluate until the data source issue is resolved.

### Recommended actions
1. Check Loki service health: `systemctl status loki` or check the Loki pod logs
2. Verify the Loki URL in the data source config matches the running instance
3. Once Loki is healthy, the alert rules should recover automatically

### Reference
- Grafana docs: Troubleshoot data source health checks
```

### Triggering AI analysis

**From the UI:**
1. Complete a test run
2. On the Run Tests page, click **Analyze with AI** (appears after run completes)
3. Analysis is appended to the current view and saved with the report

**Via API:**
AI analysis is triggered client-side by the frontend calling your configured LLM provider directly. The backend stores the analysis result alongside the report in SQLite.

## Baseline comparison for upgrade validation

A common use case is to run GrafanaProbe before and after a Grafana upgrade to detect regressions.

### Workflow

1. **Before upgrade:** Run all 17 categories and note the report ID
2. **Upgrade Grafana**
3. **After upgrade:** Run all 17 categories again
4. **Compare:**
   - Open both reports in the Reports page
   - Any category that changed from PASS → FAIL after the upgrade is a regression
   - Any new WARN in Panels or Plugins may indicate deprecated features removed in the new version

### Interpreting upgrade regressions

| Before → After | Interpretation |
|---------------|----------------|
| PASS → FAIL | Grafana upgrade broke something |
| WARN → FAIL | Pre-existing issue became critical in new version |
| FAIL → PASS | Upgrade fixed an existing issue |
| PASS → WARN | New deprecation warning in the upgraded version |

See [Upgrade Validation](../guides/upgrade-validation.md) for the complete workflow.

## Managing reports

### Delete a single report

**From the UI:**
On the Reports page, click the trash icon next to any run. A confirmation dialog appears.

**Via API:**
```
DELETE /api/reports/report-1712345678901.json
```

### Delete all reports

**From the UI:**
On the Reports page, click **Delete All** at the top. Requires confirmation.

**Via API:**
```
DELETE /api/reports
```
Returns `{ "deleted": 14 }` with the count of deleted reports.

### Storage location

| Artifact | Location |
|---------|---------|
| JSON reports | `backend/reports/*.json` |
| HTML reports | `backend/reports/*.html` |
| SQLite index | `backend/data/grafana-probe.db` |

> **Tip:** Reports are not automatically pruned. Set up a cron job or scheduled task to delete reports older than N days if storage is a concern.
